# Database Conventions

## Schema ownership

Most services own a dedicated Postgres schema; `control_plane` is the shared
control-plane store written by several API services. Cross-schema reads are
forbidden at the application layer; expose data via the streaming bus or a
dedicated API.

| Schema          | Owner service                                                                                                                                                                                                                       |
| --------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `control_plane` | Shared control-plane store (identity, policy storage F029, cost/reconciliation, routing decisions F036, analytics saved-views F038); served by `apps/api/policy-service`, `apps/api/decision-service`, `apps/api/analytics-service` |
| `gateway`       | `apps/gateway`                                                                                                                                                                                                                      |
| `audit`         | `apps/api/audit-service` (append-only ledger)                                                                                                                                                                                       |
| `scoring`       | reserved for future scoring tables                                                                                                                                                |
| `policy`        | reserved for policy-evaluation tables; policy storage (F029) lives in `control_plane`                                                                                                                               |

## Migration tooling

We use [goose](https://github.com/pressly/goose) for all SQL migrations.

- Migration files live in `platform/db/<schema>/migrations/`.
- File naming: `<5-digit-sequence>_<description>.sql` (e.g. `00002_tenants.sql`).
- Each file must have `-- +goose Up` and `-- +goose Down` blocks.
- Migrations must be idempotent (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, etc.).

### Running migrations locally

```bash
# Apply all pending migrations for control_plane
./tools/scripts/migrate.sh apply control_plane

# Dry-run (print SQL without executing)
./tools/scripts/migrate.sh dry-run control_plane

# Rollback the last migration
./tools/scripts/migrate.sh rollback control_plane

# Rehearse against a staging snapshot
./tools/scripts/migrate.sh rehearse control_plane
```

## Portability rules

- No managed-provider-only extensions (`pg_*` tied to a single cloud provider).
- Use `TIMESTAMPTZ` (not `TIMESTAMP`) for all datetime columns.
- Use `TEXT` (not `VARCHAR(n)`) for variable-length strings unless a hard limit applies.
- Row-level security (RLS) is required on all tables that carry `tenant_id`.
- No cross-service distributed transactions. Use the bus + idempotent consumers.

## Coding rules

- Every table that belongs to a tenant MUST have a `tenant_id TEXT NOT NULL` column.
- Audit tables are append-only: no `UPDATE` or `DELETE` allowed on audit rows.
- Primary keys use `UUID` (generated by the application layer, not by the DB sequence),
  so that events can be created offline or in test fixtures without a live DB.
